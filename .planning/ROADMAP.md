# Roadmap: Сервис учета оплаты ЖКУ

## Overview

v1 строится как последовательный MVP: сначала рабочий каркас с PostgreSQL, авторизацией и базовыми страницами, затем личный кабинет плательщика с квитанциями, затем публичная проверка задолженности и имитация оплаты, после этого минимальный кабинет сотрудника ЖКУ и финальная UI-полировка для дипломного показа.

## Phases

**Phase Numbering:**
- Integer phases (1, 2, 3): Planned milestone work
- Decimal phases (2.1, 2.2): Urgent insertions (marked with INSERTED)

Decimal phases appear between their surrounding integers in numeric order.

- [ ] **Phase 1: Каркас приложения и база** - Запустить full-stack основу, роли, PostgreSQL-модель и навигацию.
- [ ] **Phase 2: Кабинет плательщика и квитанции** - Дать плательщику рабочий кабинет со списком и деталями квитанций.
- [ ] **Phase 3: Проверка задолженности и оплата** - Добавить публичную проверку лицевого счета и имитацию оплаты МИР/СБП.
- [ ] **Phase 4: Кабинет сотрудника и статистика** - Добавить вход сотрудника и базовую статистику оплат.
- [ ] **Phase 5: UI-полировка и проверка v1** - Привести интерфейс и состояния к спокойному дипломному MVP.

## Phase Details

### Phase 1: Каркас приложения и база
**Goal:** Плательщик может зарегистрироваться, войти и попасть в защищенную многостраничную основу приложения, а данные хранятся в PostgreSQL.
**Mode:** mvp
**Depends on:** Nothing (first phase)
**Requirements:** [AUTH-01, AUTH-02, AUTH-04, DATA-01, DATA-02, UI-01]
**Success Criteria** (what must be TRUE):
  1. Пользователь может создать учетную запись плательщика и войти в приложение.
  2. Защищенные страницы недоступны без входа.
  3. PostgreSQL хранит пользователей, роли, лицевые счета, услуги, квитанции и оплаты.
  4. В приложении есть базовая навигация по главным страницам v1.
  5. Демонстрационные данные позволяют сразу видеть оплаченные и неоплаченные периоды.
**Plans:** 3 plans

Plans:
- [ ] 01-01: Скелет проекта React/Vite, Node.js/Express и базовая структура страниц.
- [ ] 01-02: PostgreSQL, Prisma-схема, миграции и демонстрационные данные.
- [ ] 01-03: Регистрация, вход, сессия/JWT и разделение ролей.

### Phase 2: Кабинет плательщика и квитанции
**Goal:** Плательщик видит главное меню личного кабинета, свои квитанции, детали квитанции и может изменить профиль.
**Mode:** mvp
**Depends on:** Phase 1
**Requirements:** [PAYR-01, PAYR-02, PAYR-03, PAYR-04]
**Success Criteria** (what must be TRUE):
  1. Плательщик видит сводку по задолженности и квитанциям после входа.
  2. Плательщик видит список квитанций с периодом, услугой, суммой и статусом.
  3. Плательщик может открыть конкретную квитанцию и увидеть ее детали.
  4. Плательщик может обновить основные данные профиля.
**Plans:** 3 plans

Plans:
- [ ] 02-01: API и UI главного меню плательщика.
- [ ] 02-02: API и UI списка/деталей квитанций.
- [ ] 02-03: Профиль плательщика и редактирование основных данных.

### Phase 3: Проверка задолженности и оплата
**Goal:** Посетитель может проверить задолженности по лицевому счету без входа, а плательщик может имитировать оплату квитанции через МИР или СБП.
**Mode:** mvp
**Depends on:** Phase 2
**Requirements:** [PUB-01, PUB-02, PUB-03, PUB-04, PUB-05, PMT-01, PMT-02, PMT-03, PMT-04]
**Success Criteria** (what must be TRUE):
  1. Посетитель вводит лицевой счет на главной странице и получает результат проверки.
  2. Если задолженности нет, показывается галочка и текст об отсутствии долгов.
  3. Если задолженность есть, показываются все неоплаченные периоды и виды услуг без лишних персональных данных.
  4. Плательщик может выбрать МИР или СБП на экране имитации оплаты.
  5. После имитации оплаты статус квитанции и история обновляются из базы данных.
**Plans:** 3 plans

Plans:
- [ ] 03-01: Публичная главная страница и проверка задолженности по лицевому счету.
- [ ] 03-02: Экран имитации оплаты с вариантами МИР и СБП.
- [ ] 03-03: Обновление статуса квитанции и истории оплат после имитации.

### Phase 4: Кабинет сотрудника и статистика
**Goal:** Сотрудник компании ЖКУ может войти в служебный кабинет и увидеть базовую статистику оплаченных и неоплаченных квитанций.
**Mode:** mvp
**Depends on:** Phase 3
**Requirements:** [AUTH-03, EMP-01, EMP-02, EMP-03]
**Success Criteria** (what must be TRUE):
  1. Сотрудник входит через служебную роль и попадает в отдельный кабинет.
  2. Сотрудник видит количество оплаченных и неоплаченных квитанций.
  3. Сотрудник видит суммы оплаченных и неоплаченных квитанций.
  4. Статистика меняется после имитации оплаты плательщиком.
**Plans:** 2 plans

Plans:
- [ ] 04-01: Роль сотрудника, служебный вход и защищенный маршрут.
- [ ] 04-02: API и UI базовой статистики по оплатам.

### Phase 5: UI-полировка и проверка v1
**Goal:** Пользователь получает спокойный, понятный и короткий по страницам интерфейс с корректными состояниями загрузки, ошибок, успеха и оплаты.
**Mode:** mvp
**Depends on:** Phase 4
**Requirements:** [UI-02, UI-03]
**Success Criteria** (what must be TRUE):
  1. Основные страницы используют спокойную голубую палитру, контрастные блоки и скругленные формы.
  2. Экраны не требуют длинной прокрутки для основных действий.
  3. Проверка задолженности, вход, список квитанций и оплата имеют понятные состояния ошибок и успеха.
  4. v1 можно пройти как демо-сценарий: публичная проверка -> вход плательщика -> квитанции -> оплата -> статистика сотрудника.
**Plans:** 2 plans

Plans:
- [ ] 05-01: UI-система, адаптивные стили и выравнивание страниц.
- [ ] 05-02: Демо-сценарий, состояния интерфейса и финальная проверка v1.
