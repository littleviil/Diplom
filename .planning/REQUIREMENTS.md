# Requirements: Сервис учета оплаты ЖКУ

**Defined:** 2026-05-10
**Core Value:** Плательщик должен быстро понять, есть ли у него задолженность по ЖКУ, и увидеть/оплатить свои квитанции в личном кабинете.

## v1 Requirements

Requirements for initial release. Each maps to roadmap phases.

### Authentication

- [ ] **AUTH-01**: Плательщик может зарегистрироваться с email, паролем и лицевым счетом.
- [ ] **AUTH-02**: Плательщик может войти, выйти и оставаться в своей защищенной зоне после обновления страницы.
- [ ] **AUTH-03**: Сотрудник компании ЖКУ может войти в служебный кабинет через отдельную роль.
- [ ] **AUTH-04**: Система разделяет доступ плательщика и сотрудника, чтобы пользователь не видел чужие экраны.

### Data Model

- [ ] **DATA-01**: Система хранит в PostgreSQL пользователей, роли, лицевые счета, коммунальные услуги, квитанции и записи об оплате.
- [ ] **DATA-02**: Система содержит демонстрационные данные с оплаченными и неоплаченными периодами по услугам вроде воды, газа и электричества.

### Public Check

- [ ] **PUB-01**: Посетитель может ввести лицевой счет на главной странице без авторизации.
- [ ] **PUB-02**: Если прошлые периоды оплачены, посетитель видит понятное подтверждение с галочкой и текстом об отсутствии задолженности.
- [ ] **PUB-03**: Если задолженность есть, посетитель видит все неоплаченные периоды и виды коммунальных услуг.
- [ ] **PUB-04**: Публичная проверка не раскрывает ФИО, адрес и другие лишние персональные данные.
- [ ] **PUB-05**: Главная страница кратко объясняет назначение сервиса и компании/проекта.

### Payer Cabinet

- [ ] **PAYR-01**: Плательщик видит главное меню личного кабинета с краткой сводкой по квитанциям и задолженности.
- [ ] **PAYR-02**: Плательщик видит список своих квитанций с периодом, видом услуги, суммой и статусом оплаты.
- [ ] **PAYR-03**: Плательщик может открыть детали конкретной квитанции.
- [ ] **PAYR-04**: Плательщик может изменить основные данные аккаунта в профиле.

### Payment Simulation

- [ ] **PMT-01**: Плательщик может перейти к оплате неоплаченной квитанции.
- [ ] **PMT-02**: Плательщик может выбрать имитацию оплаты через МИР или СБП с понятными визуальными иконками.
- [ ] **PMT-03**: После имитации оплаты квитанция меняет статус на "оплачено", а в базе появляется запись об оплате.
- [ ] **PMT-04**: Плательщик видит обновленную историю и статус квитанции после оплаты.

### Employee Cabinet

- [ ] **EMP-01**: Сотрудник видит служебное главное меню с базовой статистикой по оплатам.
- [ ] **EMP-02**: Сотрудник видит количество и сумму оплаченных и неоплаченных квитанций.
- [ ] **EMP-03**: Статистика сотрудника строится по фактическим статусам квитанций и оплат из базы данных.

### Interface

- [ ] **UI-01**: Приложение разделено на короткие страницы: главная, вход/регистрация, кабинет плательщика, квитанции, оплата, профиль, кабинет сотрудника.
- [ ] **UI-02**: Интерфейс использует спокойную голубую палитру, контрастные блоки, скругленные формы и понятные состояния.
- [ ] **UI-03**: Пользователь видит понятные состояния загрузки, ошибок, успешной проверки и успешной оплаты.

## v2 Requirements

Deferred to future release. Tracked but not in current roadmap.

### Companies

- **COMP-01**: Компания ЖКУ может зарегистрироваться и заполнить данные о себе.
- **COMP-02**: Сотрудник может быть привязан к конкретной компании ЖКУ.
- **COMP-03**: Компания может управлять своими видами услуг и настройками.

### Analytics

- **ANAL-01**: Сотрудник может выгружать отчет по должникам.
- **ANAL-02**: Сотрудник может фильтровать статистику по периодам, услугам и компаниям.
- **ANAL-03**: Сотрудник может экспортировать статистику в файл.

### Operations

- **OPS-01**: Сотрудник может массово загружать квитанции из файла.
- **OPS-02**: Пользователь может восстановить пароль через email.
- **OPS-03**: Пользователь может получать уведомления о новых квитанциях и задолженностях.

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Реальная платежная интеграция | Для дипломного v1 достаточно безопасной имитации оплаты |
| Полноценная регистрация компаний ЖКУ | Пользователь попросил сначала разобраться с кабинетом плательщика и квитанциями |
| Расширенные отчеты и экспорт должников | Базовая статистика сотрудника важнее для первого рабочего результата |
| Мобильное приложение | Первый релиз делается как веб-приложение |
| Публичный вывод ФИО и адреса по лицевому счету | Без авторизации нужно не раскрывать лишние персональные данные |
